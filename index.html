EC2 Summary

For this project I hosted a static website on Amazon EC2 (AWS Free Tier). I picked EC2 on purpose because it’s a real server setup: I had to create the instance, control network access, install Apache, and deploy the site like you would on an actual Linux web server.

Server stack

  • Amazon Linux
  • Apache (httpd) serving the site
  • Git for pulling updates from GitHub
  • Files hosted in /var/www/html

Security group rules

  • SSH (22) — admin access (restricted to my IP when possible)
  • HTTP (80) — public access for the website
  • HTTPS (443) — optional for future SSL

Update workflow

  1. Edit locally
  2. Commit + push to GitHub
  3. On EC2: git pull

The main takeaway is that EC2 forces you to understand the basics: networking rules, Linux administration, and how deployments work without “one click” hosting.

Project Summary

This project demonstrates how to deploy a static website on AWS using a traditional server approach. I launched an EC2 instance (Free Tier), installed Apache, and served the site from /var/www/html. The website code is stored in GitHub and deployed to the server using Git.

  • AWS EC2 (Free Tier)
  • Apache HTTP Server
  • Git + GitHub
  • Security Groups (Firewall Rules)
  • Linux Commands + Troubleshooting
  • Repeatable Update Process

Architecture

Traffic flow:

  1. User visits the EC2 Public IPv4 address in a browser
  2. AWS Security Group allows inbound HTTP (80) (and optionally HTTPS (443))
  3. Apache serves the static files from /var/www/html

This version is intentionally simple: no database and no backend services. The goal was to learn EC2 server setup, networking rules, and deployment workflow.

GitHub Setup and Issues We Fixed

I created a GitHub repository and pushed the project files (index.html, assets/, images/) so the EC2 instance could pull them.

Problems we ran into

  • “git is not recognized” — Git wasn’t installed on the local machine.
  • “Please tell me who you are” — Git identity wasn’t configured, so commits failed.
  • Push/authentication issues — fixed by logging in properly (token/auth method depends on setup).

AWS EC2 Setup and Security

I launched an EC2 instance (Amazon Linux) and used a security group to control inbound traffic. The security group works like a firewall.

Inbound rules

  • SSH (22) — used to connect/administer the instance
  • HTTP (80) — required so the website loads
  • HTTPS (443) — optional for SSL later

SSH connection issue

Even when the instance showed “checks passed,” we hit SSH connection errors. The fix was making sure the security group rule matched the way we were connecting.

  • When using EC2 Instance Connect, restricting SSH to “My IP” can block the browser connection.
  • We temporarily allowed SSH from Anywhere IPv4 to get in, then locked it back down.

Apache Install and Deployment

After connecting to the instance, I installed Apache and Git, enabled the service, and deployed the site by cloning the repo. 

sudo yum update -y
sudo yum install httpd git -y
sudo systemctl enable --now httpd

cd /var/www/html
sudo rm -f index.html
sudo git clone https://github.com/damurphy1/aws-final-project.git .

The dot at the end of the clone command puts the repository contents directly into /var/www/html.

Updating the Website

Update process:

  1. Edit locally
  2. Push to GitHub
  3. Pull on EC2

Local

git add .
git commit -m "Update website content"
git push

EC2

cd /var/www/html
git pull

Problems we fixed during updates

  • Dubious ownership — fixed by correcting ownership and/or marking the directory as safe: 
    sudo chown -R ec2-user:ec2-user /var/www/html
git config --global --add safe.directory /var/www/html
  • Local changes would be overwritten — fixed by stashing or resetting: 
    git stash
git pull
    git reset --hard
git pull
